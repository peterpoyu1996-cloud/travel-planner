# MVP 功能清單｜旅遊決策助手（沖繩試點）v0.1

優先級定義：**P0** = 沒有它 demo 就不成立；**P1** = 明顯提升體驗，但可延後；**P2/Out** = 明確排除，列出是為了避免範圍蔓延。

## P0（必須有，MVP 核心）

| # | 功能 | 說明 |
|---|---|---|
| 1 | 條件輸入表單 | 日期、住宿地、租車、有無小孩＋年齡、預算、出發地 |
| 2 | 沖繩景點/飯店/餐廳知識庫（RAG） | 80–150 筆景點/餐廳＋20–30 筆飯店，含親子適合度、雨天可用性、預算區間標籤 |
| 3 | 多日行程生成 | 依條件產出逐日景點/餐廳建議，含推薦原因 |
| 4 | 地理合理性排序 | 避免同一天景點分散在本島兩端來回跑（技術風險最高，優先驗證） |
| 5 | 行程檢視頁面 | 依天分頁/卡片呈現，含推薦原因文字 |
| 6 | 單點「換一個」 | 對候選池重新推薦替代景點 |
| 7 | 候選不足的 fallback 提示 | 避免空結果或明顯不合理的湊數行程 |

## P1（第二輪加分項，MVP 驗證成功後再做）

| # | 功能 | 說明 |
|---|---|---|
| 8 | 行程匯出（PDF/文字） | 方便帶出去用 |
| 9 | 手動新增/刪除景點 | 限知識庫已收錄範圍內 |
| 10 | 租車 vs 大眾運輸提示 | 靜態建議文字，非即時導航 |
| 11 | 行程分享連結 | 不需帳號系統，用唯一連結即可 |

## P2 / Out of Scope（明確不做，避免範圍蔓延）

| # | 功能 | 排除原因 |
|---|---|---|
| 12 | 天氣/營業時間即時感知與改建議 | 這是原始構想的核心，但依賴多個即時資料源＋動態演算法，風險過高，列為 Phase 2 |
| 13 | 即時定位＋主動推播 | 需要通知權限、背景定位，屬 Phase 3 |
| 14 | 使用者帳號系統 | MVP 用 session/連結暫存即可驗證核心價值 |
| 15 | 訂房/訂餐/租車下單串接 | 屬於交易層功能，與「決策建議」核心價值無關，且開發成本高 |
| 16 | 多語系介面（英/日） | 先驗證中文使用者的核心體驗 |
| 17 | 石垣島/宮古島等離島 | 先聚焦沖繩本島，資料量與地理複雜度可控 |

## 驗證 MVP 是否成功的判準

不追求商業指標，聚焦「核心決策邏輯是否站得住腳」：

1. 用 5–10 組不同條件組合（帶小孩/不帶、租車/不租、不同預算）跑過一輪，行程是否**地理合理**（不會同一天在本島南北來回跑）
2. 每個推薦是否都有**站得住腳的推薦原因**，而不是通用套話
3. 「換一個」功能是否真的能給出**同樣符合條件**的替代方案，而非隨便換一個不相關的點
4. 端到端流程（輸入→生成→檢視→換一個→匯出）**不 crash、不卡死**，15 秒內出結果

## 下一步建議

1. 先手動整理 20–30 筆沖繩重點景點的資料（含人工標籤），做一個「假資料版」demo，驗證 Prompt 與生成邏輯，不急著做完整 API 串接
2. 地理分群/動線合理性這個技術風險，建議先用簡單規則（例如依地區分群：那霸市區／中部／北部）而不是全靠 LLM 自由發揮
3. 確認 Google Places API 使用條款是否允許你的資料儲存/展示方式，避免後期重做
