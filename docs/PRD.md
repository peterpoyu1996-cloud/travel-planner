# PRD｜旅遊決策助手（沖繩試點）v0.1 - MVP

## 1. 產品概述

**一句話定位**：一個讓使用者輸入旅遊條件（日期、住宿地、是否租車、是否帶小孩、預算、目前位置），由 AI 基於沖繩景點/飯店/餐廳知識庫，自動生成多日行程建議的旅遊決策助手。

**產品願景（North Star，非 MVP 範圍）**：未來能根據天氣、時間、距離、景點營業時間等即時因子，主動建議「今天要不要改行程」。MVP 先不做即時感知，聚焦在把「根據條件生成靠譜行程」這件事做扎實。

**專案定位**：驗證構想的個人專案/作品集，非以商業化為首要目標。PRD 因此不含定價、訂閱等商業模式章節；但會保留未來擴充的路徑。

## 2. 問題陳述

旅遊規劃者（尤其是帶小孩、需要租車的家庭旅遊者）在規劃沖繩多日行程時，需要同時比對：景點與住宿的地理距離、是否適合小孩、租車動線是否順、預算分配是否合理。現行做法是在多個網站/論壇/Google Maps 之間手動比對，耗時且容易漏掉「距離太遠」「不適合小孩」這類隱性限制。

## 3. 目標使用者

- **主要人物誌**：計劃帶小孩去沖繩自由行、會租車、對行程規劃沒有太多時間研究的家庭旅遊者。
- **次要人物誌**：不帶小孩、但想要「輸入條件就拿到可用行程」的自由行旅客。

## 4. 範圍與限制（重要）

| 維度 | MVP 範圍 |
|---|---|
| 地理範圍 | 僅沖繩本島（先不含離島如石垣島、宮古島，除非資料驗證後有餘裕） |
| 行程長度 | 支援多日（例如 3–7 天），但**不做逐小時排程**，只做「每天建議去哪幾個點＋建議順序」 |
| 即時因子 | **不做**天氣/營業時間/即時距離的動態感知與改建議（列入 Phase 2） |
| 語言 | 使用者介面與輸出為繁體中文；知識庫來源可能含日文資料，需在後端轉譯/摘要為中文 |
| 租車建議 | 提供「是否需要國際駕照」「建議租車 vs 大眾運輸」的靜態提示，不做即時路況/導航 |

### 明確不做（Out of Scope）
- 訂房/訂餐/租車的實際下單或串接
- 使用者帳號系統、多人協作行程（v0.1 可先不做登入，用 session 暫存）
- 即時定位追蹤與主動推播
- 多語系（英文/日文介面）

## 5. 功能需求（Functional Requirements）

### FR1. 條件輸入表單
使用者需輸入：
- 出發/回程日期（決定天數）
- 住宿地（地址或飯店名稱，可用文字輸入＋地圖選點）
- 是否租車（是/否）
- 是否帶小孩（是/否；若是，需追問年齡區間，因為景點適合度會依年齡不同）
- 預算（每日或總預算，區間選擇即可，不需精確數字）
- 目前所在地（v0.1 可簡化為「出發地」，例如「那霸機場」，非即時定位）

### FR2. 行程生成（RAG 驅動）
- 系統從知識庫中，依「地理位置分群＋親子適合度＋預算區間＋租車與否」篩選候選景點/餐廳/飯店
- LLM 依篩選結果與使用者條件，生成逐日行程建議（每天 3–5 個景點/餐廳，含建議停留時間、彼此距離提示）
- 每個推薦項目需附上「推薦原因」（例如：「適合帶小孩、車程 20 分鐘、雨天也可室內活動」），建立使用者對 AI 建議的信任

### FR3. 行程檢視與調整
- 使用者可依天檢視行程卡片列表
- 使用者可對單一景點按「不喜歡/換一個」，系統重新從候選池推薦替代方案（同樣需符合親子/預算/距離條件）
- 使用者可手動刪除/新增景點（v0.1 新增可先限制在知識庫已收錄範圍內）

### FR4. 行程總覽輸出
- 產出可分享/匯出的行程總表（初期可為單頁網頁檢視 + 簡易 PDF/文字匯出）

## 6. 資料需求（RAG 知識庫）

**收錄範圍**：沖繩本島熱門景點、飯店、餐廳。

**資料來源策略**：混合模式
1. **API 打底**：透過 Google Places API（或同等服務）取得基礎資料——名稱、地址、座標、營業時間、評分、照片、基礎分類
2. **人工精選補強**：針對重點景點/餐廳（例如美麗海水族館、國際通、備瀨福木林道等），人工撰寫「適合親子與否」「建議停留時間」「雨天備案適用性」等 API 無法直接取得的標籤

**每筆資料建議欄位**：

> 以下欄位設計參考使用者過往實際規劃的沖繩行程表（2024/1/7–1/11），從中萃取出規劃時真正在意、但一般景點資料庫（如 Google Places）不會提供的欄位。原始行程表格式每天不同（分開查、分開記）是正常的原始資料型態，代表規劃邏輯一致但未結構化，以下標籤即是把這套邏輯結構化的結果。

**A. 基本資訊**
- 名稱、地址、座標、分類（景點/飯店/餐廳）、營業時間、官網/電話、評分

**B. 動線與交通（Movement）**——證據：「8點半出發到海中觀景塔｜大約9:30到」「18:30出發到餐廳｜30分鐘車程」
- `travel_time_from_prev`：與上一站/常見出發點的車程或交通時間（分鐘，可為區間）
- `travel_mode`：開車／步行／大眾運輸
- `mapcode`：沖繩租車導航必備欄位，一般地址型 API 不提供，需另外查詢/轉換
- `suggested_stay_duration`：建議停留時間

**C. 停車（Parking）**——證據：「停北口停車場（P9停車場進去直走停P7立體停車場）」「若狹海濱公園停車場 33 186 181」（停車場與景點本身 MapCode 常不同）
- `has_parking`（布林）
- `parking_name`：停車場名稱
- `parking_mapcode`：停車場專屬 MapCode（易被忽略但實務上很關鍵）
- `parking_notes`：轉乘/樓層/免費或付費等特殊說明

**D. 備案與彈性設計（Backup / Flexibility）**——證據：「在房間耍廢到時間到(泡溫泉~)」（刻意留白的緩衝時段）、「有需要的話再去(TOM PLAZA)」（條件式行程）、涮涮鍋「需先預訂」
- `is_flexible_slot`：是否為刻意設計的彈性/緩衝時段（不是必去景點，是行程節奏的緩衝）
- `conditional_note`：條件式描述（例：有需要才去）
- `requires_reservation`：是否需預約（沒預約成功時需要有備案）
- `backup_option`：替代方案，可連結到知識庫中另一筆資料的 ID

**E. 決策相關標籤（人工/半自動標註）**
- 適合親子（是/否＋年齡建議）、室內或戶外（雨天可用性）、預算區間（$/$$/$$$）、鄰近景點群組（供地理分群）

**F. 向量化用文字描述**：給 Embedding 使用，融合以上 A–E 欄位生成一段自然語言描述

**資料規模估算（MVP）**：建議先收錄 80–150 個景點/餐廳、20–30 個飯店，足以支撐 3–7 天行程且不重複度尚可接受。

**資料萃取方式的延伸思考**：使用者原始研究習慣是「先查好 A→B 車程＋停車＋要不要預約，再排時段」，這與本產品要提供的價值高度重疊。未來可考慮讓使用者貼上自己查好的原始筆記（格式不拘），由 AI 自動萃取並對應到上述欄位，而不需要使用者手動填表——但此屬 Phase 2+ 範圍，MVP 階段仍以人工/半自動方式建立知識庫。

## 7. 非功能需求

- **回應時間**：單次行程生成應在 15 秒內完成（含 RAG 檢索＋LLM 生成）
- **可維護性**：知識庫需可透過後台/腳本手動更新，不需重新部署程式
- **成本控制**：需記錄 LLM API 呼叫成本（token 用量），避免單次生成成本失控
- **資料新鮮度**：營業時間/是否歇業等資訊需有更新機制（v0.1 可接受手動每季度檢查）

## 8. 技術架構建議

- **前端**：網頁 App（Next.js 或同等 SPA 框架）
- **後端**：RAG pipeline（向量資料庫，如 pgvector/Chroma/Pinecone 皆可，依團隊熟悉度選擇）＋ LLM（Claude）做生成與推理
- **資料前處理**：Google Places API 抓取 → 清洗/補標籤 → Embedding → 存入向量庫
- **儲存**：使用者輸入與生成行程可先用瀏覽器 session/簡單資料庫暫存，不需完整帳號系統

## 9. 成功指標（作品集/驗證階段指標）

- 能否針對 10 組不同輸入條件（親子/無小孩、租車/不租車、不同預算），穩定生成「地理上合理、不重複、符合條件」的多日行程
- 生成行程中「推薦原因」的準確度與可信度（人工抽查）
- 端到端 demo 可用，無明顯 crash 或空結果

## 10. 風險與待決事項

- **資料授權風險**：Google Places API 有使用條款限制（例如資料快取時效、禁止大量離線儲存），需在資料前處理階段確認合規做法
- **多日行程的地理動線複雜度**：若景點分散在沖繩本島南北，天數分配演算法需要有基本的地理分群邏輯，否則行程會「來回跑」，這是 MVP 階段最大的技術風險，建議優先驗證
- **親子適合度標籤的主觀性**：需要定義清楚的標註標準，避免 AI 生成不一致的建議

## 11. Roadmap（後續階段，供對齊方向用）

- **Phase 2**：加入天氣 API／即時營業狀態，對已生成行程做「衝突提示」（使用者手動觸發重新規劃）
- **Phase 3**：即時定位＋主動推播「現在建議改去哪」，達成最初設想的即時決策助手
