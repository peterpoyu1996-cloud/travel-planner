# ROADMAP

這份文件回答「現在做到哪、接下來做什麼、以後才做什麼」，時間軸/優先順序導向。
跟 [MVP.md](MVP.md) 不同：MVP.md 是「這個版本該不該做這個功能」的範圍清單，
ROADMAP.md 是「這些已同意的功能，大概什麼順序上線」。

## 維護方式

- 每完成一個里程碑（不是每個 commit）才更新這份文件
- 狀態只有三種：✅ 已完成／🚧 進行中／⏳ 未開始，不用寫百分比進度
- 順序調整、優先權改變時要更新，並在 [CHANGELOG.md](../CHANGELOG.md) 留一筆記錄原因

## 資深規劃師缺口評估（2026-07-18 討論）

以「這樣的資料夠不夠拼出各式人們需要的行程」為標準做的評估，完整討論見對話紀錄，結論摘要：

**現在的資料拼得出來**：北部沿海景點＋開車＋親子取向的行程（因為 Excel 種子資料剛好是這個取向）。

**現在的資料拼不出來（結構性缺口，不是量不夠）**：
1. 不開車的行程（北部/中部完全沒有大眾運輸資料）—— 🚧 部分處理中，見上方「不開車行程地基」
2. 玩水行程（沒有海灘安全性/設施資訊）—— 🚧 部分處理中，補了海灘廣度，深度還沒
3. 多代同堂/無障礙行程（完全沒有無障礙欄位，查過的 `okinawa-bf-map.jp` 可信度存疑，SSL憑證都設定錯）—— ⏳ 未開始，目前沒找到夠好的資料源
4. 精算預算（budget_level 只有 $/$$/$$$ 桶分類，沒有實際票價數字）—— ⏳ 未開始
5. 點到點交通時間（只有 haversine 直線距離，不是真實路況時間；已用於單軌步行判斷，但景點之間的車程還是沒有）—— ⏳ 未開始
6. 季節性資訊（颱風季/水溫/賞鯨季，現在的資料完全沒有時間維度）—— ⏳ 未開始
7. 離島跳島（慶良間諸島等，本來就不在 PRD 範圍內）—— 明確不在近期規劃
| `enrich_llm.py` 實際跑一輪標註親子適合度等欄位 | ⏳ | 需 API key，成本已估算 <$1 |
| 知識庫規模擴充到 80-150 筆 | ⏳ | 對應 PRD 資料規模估算 |
