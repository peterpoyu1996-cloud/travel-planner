## region_group 分界（緯度，沖繩本島）

三分法邊界跟 `scraper/merge_osm.py`／`scraper/ingest_osm.py` 用的是同一組，改一邊要記得改另一邊：

| region_group | 緯度範圍 | 大致對應 |
|---|---|---|
| 北部 | lat ≥ 26.45 | 名護、本部、國頭、古宇利島 |
| 中部 | 26.25 ≤ lat < 26.45 | 沖繩市、北谷（美國村）、讀谷、嘉手納 |
| 南部 | lat < 26.25 | 那霸、瀨長島、系滿、南城 |

只檢查緯度（南北軸），沖繩本島狹長，緯度已足夠區分；資料沒有 lat 時無法自動分類，需人工標註。

## 資料現況

- `attractions.json` / `hotels.json` / `restaurants.json` 有兩種來源：
  - `source: "excel_seed"`：使用者 2024/1/7–1/11 沖繩行程 Excel 轉錄的種子資料，核心欄位（MapCode/停車/備案）最完整，但只有 15 筆
  - `source: "osm"`：`scraper/ingest_osm.py` 從 OpenStreetMap 開放資料匯入，補廣度，但只有基本欄位（名稱/座標/分類/部分營業時間），MapCode/停車/親子適合度等核心決策欄位是 `null`，需要之後人工或 LLM 補標註
- `lat`/`lng`/`rating` 等欄位沒有來源就標 `null`，待後續跑 geocoding 或官網查證再補，**不可用猜測值填入**
- 原始 Excel 檔案是個人行程資料，**刻意不進版控**（見 `.gitignore` 的 `*.xlsx`），只有轉錄、去識別化後的 JSON 才進 repo
- 下一步：針對 `source: "osm"` 的資料，優先幫熱門項目補 MapCode 與親子/預算標籤（`scraper/enrich_llm.py`）

## name 欄位繁體中文化現況

`scraper/convert_traditional.py`（機械式新字體→繁體字元替換，零風險）+ `scraper/translate_names.py`
（LLM 輔助翻譯，只翻有把握的知名連鎖店/地標）處理過 150 筆 OSM 資料，現況：

- 45 筆本來就是純漢字、字元剛好跟繁體中文相同，不用改
- 46 筆已轉換/翻譯成繁體中文（含機械轉換、LLM翻譯、OSM社群提供的 `name:zh`/`name:zh-Hant` 標籤）
- 59 筆含假名（片假名/平假名），`name` 沒有把握正確翻譯，**刻意保留日文原名**（`name` 等於 `name_local`），
  這是刻意的取捨，避免翻錯誤導使用者，不是漏做

⚠️ **踩過的坑**：OSM 社群提供的 `name:zh`/`name:zh-Hant` 標籤不是自動信任的來源，發現過至少 2 筆
明顯的形近字誤植（例如「岸本食堂」被標成「案本食堂」、「花笠食堂」被標成「花苙」）。已人工訂正這兩筆，
但沒有逐筆全部覆核，之後若要大量信任 OSM 的中文標籤，建議先抽樣驗證。

## translated_name 欄位（`scraper/add_translated_name.py`）

因為前台顯示不該出現「還沒翻譯的日文」，所以另外加了 `translated_name` 欄位，
一定是中文，跟上面謹慎保守的 `name` 欄位分開：

- 上面那 91 筆（45+46）`name` 已經是中文的，`translated_name` = `name`，兩欄位一樣
- 剩下 59 筆 `name` 還是日文的，`translated_name` 放 LLM 輔助翻譯結果（`name` 保持不動）

這批翻譯裡有不少沖繩方言詞彙（バンタ、ゆいまーる、ぶくぶく、なかゆくい 這類），
是音譯兼意譯，**不保證跟當地慣用譯名一致**，信心程度比「name 已經是中文」那 91 筆低。
要判斷一筆資料的翻譯可不可靠，比較 `name` 是否等於 `name_local` 就知道：
不相等 = 這筆 `translated_name` 是新翻的、信心較低。

## 不開車行程的地基（`scraper/geo_utils.py`、`data/transit_stations.json`）

針對「不開車行程幾乎拼不出來」的問題（見 [docs/ROADMAP.md](../docs/ROADMAP.md) 的評估），
先做了三件低成本、高確定性的事：

1. **`scraper/geo_utils.py`**：haversine 直線距離工具，純數學公式零成本，供其他腳本共用
2. **`data/transit_stations.json`**：那霸單軌電車（ゆいレール）全 19 站座標，抓自 OSM
   `railway=station`（沖繩本島只有這一條軌道系統，查詢範圍內的站點不需要額外篩選）
3. **`scraper/tag_transit_access.py`**：幫每筆資料算到最近單軌站的直線距離，1.2km 內視為
   步行可達，寫入 `nearest_monorail_station`／`monorail_distance_km`。`backend/app/filters.py`
   的 `has_car=false` 篩選邏輯已經會用這個欄位放行「標記開車但其實走得到單軌站」的資料

**已知限制**：這只解決了那霸市區周邊的不開車需求。北部/中部本來就沒有單軌，需要巴士資料才能解，
查過沖繩有 GTFS 巴士開放資料生態（OTTOP），但還沒確認北部/中部主要巴士公司（沖縄バス/琉球バス交通/
那覇バス/東陽バス）是否有涵蓋，這塊還沒做。

## 玩水景點（`scraper/ingest_osm.py` 的 beach 類別）

原本的 OSM 查詢完全沒抓 `natural=beach`，等於本島大部分實際海灘都不在知識庫裡。已經補查詢、
匯入 25 筆命名海灘到 `attractions.json`（id 前綴 `osm-beach-`）。這些海灘目前只有名稱/座標/地區，
**沒有「適合小孩/有救生員/有淋浴設施」這類決策資訊**，那部分還是得靠人工查證官網/部落格，
沒有捷徑，見 docs/ROADMAP.md。

## 高速公路感知的車程估算（`scraper/highway_routing.py`、`scraper/travel_time.py`）

原本的 `travel_time_from_prev` 只有 Excel 種子資料當初實際那次的紀錄，`monorail_distance_km`
也只是單點到單點的直線距離，都沒辦法回答「這兩點之間開車大概多久、會不會經過高速公路」。
新增三支腳本組成一條處理鏈：

1. **抓資料**：`scraper/sources/osm_overpass.py` 新增 `highway_geometry`（沖縄自動車道等
   motorway/motorway_link 路段幾何）、`highway_junctions`（交流道節點）兩個查詢類別
2. **`scraper/build_highway_network.py`**：整理成 `data/highway_network.json`——路段幾何
   （畫圖用）＋ 27 個交流道（同名節點，例如同一個交流道的進/出匝道，合併成一個代表點）
3. **`scraper/highway_routing.py`**：把路段幾何拆成圖（節點=座標、邊=相鄰點的haversine距離），
   用 Dijkstra 算任兩個交流道之間「沿著公路網實際開多遠」，不是交流道座標直接算直線距離
   （沖縄自動車道有彎、有分支，直線距離會低估）
4. **`scraper/travel_time.py`**：`estimate_minutes()` 比較「直接走平面道路」跟「開到最近交流道
   上高速、開到最近交流道下高速、再開到目的地」兩種算法，取比較快的，回傳用了哪條路線。
   速度假設：高速公路 90km/h、平面道路 45km/h（含1.3倍繞路係數修正haversine低估）

**已知限制**：
- 交流道之間偶爾在 OSM 資料裡沒連通（例如支線道路的連接路段沒被 `motorway`/`motorway_link`
  標籤查到），`estimate_minutes()` 用「試離起訖點最近的5個交流道各種組合」來繞過大部分這種情況，
  但不保證100%找得到路，找不到就會誤判成「這段路沒有更快的高速公路選項」而非真的沒有
- 速度假設是固定值，沒有考慮車流/紅綠燈/時段
- 已接進 `backend/app/itinerary.py`（`fill_real_travel_times()`），行程生成後會用候選資料的真實座標覆蓋 LLM 填的 `travel_time_from_prev`，涵蓋同一天內相鄰站點，也涵蓋跨天（前一晚住宿到隔天第一站）

## 決策相關欄位補標（2026-07-18 產品/架構/旅遊規劃三方討論後的執行）

針對「資料夠不夠拼出行程」的評估發現 `budget_level`（191筆僅1筆有值）、`kid_friendly`
（191筆僅3筆是true）、`mapcode`/`has_parking`（皆91% null）這幾個核心決策欄位幾乎等於沒填，
對應執行了以下補值：

- **`kid_friendly`**：由 AI（Claude Code 本身，不是另外呼叫付費 API 的 `enrich_llm.py`）
  依景點/餐廳「類型」逐筆判斷（動物園/水族館/樂園/公園/命名沙灘/家庭連鎖餐廳→true；
  酒場/Bar/釀酒廠導覽→false；不確定的一律維持 null，不硬猜），標記 `source: llm_enriched`，
  `source_note` 保留原始來源說明。戰爭主題資料館（沖繩縣平和祈念資料館、姬百合資料館、
  舊海軍司令部壕）刻意不設 `kid_friendly=false`（不是「不適合」，是「內容沉重、要看年齡」，
  二元欄位講不清楚），改填 `kid_age_notes` 說明。填值後：true 98筆／false 7筆／null 86筆
  （原本 true 只有3筆）
- **25筆海灘**：先全部标 `kid_friendly=true`（依「命名沙灘、OSM natural=beach」的類型判斷），
  之後針對其中16筆額外用 WebSearch 查證監視員/防水母網/淋浴/廁所等實際安全設施資料，
  查到「沒有監視員/沒有防護網/歷史上有鯊魚出沒紀錄/官網明講不適合游泳」的11筆，把
  `kid_friendly` 從最初的 true 改回 null 或 false，並在新用到的 `conditional_note`
  欄位寫清楚具體原因（例：「⚠️沒有監視員、沒有防水母網，曾有鯊魚目擊紀錄...」）。
  這是刻意的自我修正：類型判斷（是海灘＝大概率親子友善）跟實際安全資料（有沒有人看顧）
  是兩回事，查到後者就該覆蓋前者的粗略推測
- **`budget_level`**：僅對「有真實價格根據」的餐廳補值——知名連鎖店（CoCo壱番屋、牛角、
  串カツ田中、Jolly Pasta、金沢まいもん寿司、Royal Host、A&W、Steak House 88）用 WebSearch
  查證當前一般預算區間；另外兩筆知識庫既有餐廳（燒肉本舗 島牛、燒肉食堂 Kouya）查到官網
  自述「お手頃価格」與實際套餐價（2980日圓）佐證。沒有真實價格依據的其餘餐廳/飯店維持
  null，**沒有靠關鍵字或分類瞎猜價格**（這是「事實類」欄位，猜錯的成本比 kid_friendly
  這種類型判斷高很多）。填值後：11/191 有值（原本1/191）
- **`mapcode`/`has_parking`/`parking_notes`**：針對兩個範例情境實際會用到的候選（首里城公園、
  燒肉本舗島牛、Smile飯店那霸城市度假村、蒙特雷飯店沖繩溫泉度假村、Hilton Okinawa Sesoko
  Resort、燒肉食堂Kouya、東橫INN那霸新都心Omoromachi）逐筆用 WebSearch/WebFetch 查證官網，
  補齊停車場資訊；MapCode 只有 Hilton Sesoko 查到官方公布的代碼（206 823 457*17），其餘
  官網/一般搜尋都沒有公開 MapCode，誠實留 null 並在 `source_note` 註明已查過、不是漏做

### `rating` / `description_for_embedding` / `backup_option` / `is_flexible_slot` 去留評估

這四個欄位填值率都接近或等於 0%，評估後決定**全部保留、不從 schema 移除**：

- **`rating`**：無法合規補值——Google/Tripadvisor 等有評分資料的來源都是 C 級（見
  ARCHITECTURE.md 第4節，明文禁止爬蟲），沒有 A/B 級開放來源提供結構化評分。這不是
  「忘記做」，是結構性缺口，移除欄位也解決不了問題，保留給未來人工/使用者評分機制用
- **`description_for_embedding`**：原本是為向量 RAG 設計的欄位，但架構已經決定改用規則式
  過濾（見 ARCHITECTURE.md 第1節），這個欄位的「向量化」原始用途已經沒有對應的實作。
  不過它作為「給 LLM 的額外自由文字描述」還是有價值——`itinerary.py` 的 `build_user_prompt()`
  會把整個 candidate dict 塞進 prompt，任何欄位都會變成 LLM 看得到的上下文。保留欄位，
  但更新用途說明：**這欄位現在的實際功能是「給 LLM 的補充描述」，不是「等著做向量搜尋」**
- **`backup_option`／`is_flexible_slot`**：對應 PRD 提到的「備案與彈性設計」，這次會期內
  新用到了同一組欄位裡的 `conditional_note`（幫16筆海灘寫安全注意事項），證明這組欄位設計
  是有用的，只是還沒有人工逐筆填值。保留，列為下一步可以延伸的工作（例：幫熱門景點標記
  「雨天備案」「客滿時可以去哪裡」），不是要拿掉

## 官方開放資料匯入（2026-07-18，Plan D 第一層）

### 沖縄県観光施設一覧（CC-BY 4.0）

來源：`data.bodik.jp/dataset/churashisetu_okinawa-dpf_`，2024年2月時點快照，998筆裡
**只有16筆填了完整欄位**（其餘是空白列，判斷是政府匯出時的格式殘留，不是刻意隱藏資料）。

## kid_friendly 星等制（2026-07-18，Plan D 第二層：部落格交叉查證試點）

### 為什麼從布林值改成星等

產品/AI架構/旅遊規劃/測試/資料治理/內容研究六方會議的結論：`kid_friendly` 原本的
`true/false/null` 三態，把「這個景點對帶小孩到底好不好」這種本質上是**分歧、機率性**的
現實資訊，硬壓成二選一，資訊損失很大（例如「有些小孩看YouTube影片崩潰大哭，但也有很多
小孩玩得很開心」這種真實但矛盾的部落格/影音回饋，boolean 完全表達不出來）。改成 **0-5 星
（0.1級距）+ null**，null 代表「還沒查證到足夠證據」，不是「不適合」。

級距從最初的 0.5 改成 0.1（2026-07-18 使用者回饋「0.5太粗」後調整）：同一批證據強度不同的
案例（例如同樣落在「高度正面」區間，但其中一筆有更多摩擦點）現在可以被區分開，不用被迫
綁在同一個粗略級距上。要注意的是，0.1 級距不代表判斷精確到小數點第一位——**這仍然是質化
判斷後的相對排序，不是量化公式算出來的**，`source_note` 裡的判斷依據才是重點，星等數字本身
只是排序用的摘要，不要誤讀成「4.4分跟4.3分有統計上顯著的差異」。

### 資料品質門檻（避免星等淪為另一種猜測）

- 每筆星等**必須查證到至少3篇獨立網域**的部落格/遊記交叉比對才能填值，少於3篇一律維持
  null，並在 `conditional_note` 或 `source_note` 寫「查無足夠交叉驗證資料」
- 每篇查證來源要記錄網域、發布年份，寫進 `source_note`，可追溯，不能只寫「部落客普遍反映」
  這種無法查核的話
- 星等評分邏輯：以「基準3分」為中性起點，每篇來源的正面/負面/結構性限制（例如無障礙/
  嬰兒車友善度）訊號人工綜合判斷後調整，不是套用死板公式硬算——這是質化判斷，`source_note`
  裡會寫清楚判斷依據，讓之後的人（或未來的自己）能檢查這個判斷合不合理
- **絕對不把舊的 true/false 機械轉換成星等**（例如 true→4.0），那本身就是猜測，違反
  CLAUDE.md 的資料完整性硬性規則。舊值原封不動保留在 `kid_friendly_legacy_flag`，星等只在
  真的重新查證後才填入新欄位

## recommendation_score（一般推薦度，2026-07-18新增）

### 跟 `rating` 的關係——刻意分成兩個欄位，不是重複

`rating` 欄位維持之前的評估結論：**結構性缺口**，因為有結構化評分數字的來源
（Google Maps／Tripadvisor／食べログ等）全部是 C 級，明文禁止爬蟲，合規來源查不到
現成的評分數字，所以 `rating` 會繼續是 0% 填值率，這不是失職，是資料源本身不存在。

`recommendation_score` 是完全不同的東西：**我們自己用合規的部落格/評論網站內容，
交叉比對出來的質化推薦度分數**，不是抓取別人的評分系統，是自己判斷、自己負責。
跟 `kid_friendly` 星等制同一套方法論（0-5星、0.1級距、需≥3篇獨立來源、絕不硬猜），
但衡量的維度不同——`kid_friendly` 問的是「帶小孩去好不好」，`recommendation_score`
問的是「一般人值不值得去」，兩者刻意允許分數不同，而且這次試點也證實了兩者真的會分歧：

| id | kid_friendly | recommendation_score | 分歧原因 |
|---|---|---|---|

這張表本身就是這個雙欄位設計有沒有意義的證據——如果兩欄位永遠分數一樣，那其中一個是多餘的。

### 查證方法（沿用 kid_friendly 的門檻，不重複規定）

一樣要求 ≥3 篇獨立來源、一樣不做舊資料機械轉換（`rating` 本來就是 null，沒有舊值可轉）、
一樣把爬蟲來源合規分級套用在部落格/評論網站上。差別是這次查證時刻意**不用「親子」相關
關鍵字**，改用「評価／口コミ／おすすめ度／がっかり／微妙／つまらない」這類中性或負面
關鍵字去查，避免查證結果被親子情境的既有結論汙染，確保這是獨立判斷出來的分數。
